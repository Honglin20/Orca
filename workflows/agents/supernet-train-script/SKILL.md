# Supernet Training Script Generator

Use this skill to generate supernet training scripts and a shell launcher for a NAS supernet that was previously generated by the `pytorch-model-optimizer` skill. This skill reads the upstream summary, analyzes the user's training code, determines supernet training viability, generates the training script and launcher, and completes the supernet training summary.

Skill resource paths:

- `<skill_dir>`: The directory containing this `SKILL.md` file. All `references/` paths in this skill are relative to `<skill_dir>`.
- Do NOT read files under `<skill_dir>/references/workflow-checklists/`. Those are exclusively consumed by the `workflow-verifier` subagent.

## Lazy Loading

Do **not** read all reference files, workflow documents, or asset files upfront. Only read the materials that a specific step requires **when you begin that step**. This keeps the context focused and avoids loading irrelevant content that may never be needed.

## Required Inputs

- `<output_dir>`: Directory containing the generated supernet artifacts. Must contain `supernet.py`, `inspect_supernet.py`, and `supernet_summary.md`.
- `<user_project_root>`: The root directory of the user-provided PyTorch project. If not explicitly provided, read from the **Source Project** section of `supernet_summary.md`.

## Working Directory and Path Conventions

- `<output_dir>` **(working directory)**: All generated artifacts are written under `<output_dir>`. **Run `cd <output_dir>` once before executing any commands in this skill**; the working directory persists across subsequent commands, so all relative paths (e.g. `run_train_supernet.sh`, `train_supernet.py`) will resolve correctly. Sibling modules (e.g. `supernet.py`) are importable as plain imports without `sys.path` or `PYTHONPATH` manipulation. File paths in the steps below are relative to `<output_dir>` unless otherwise noted.
- `<nas_agent_root>`: The root directory of the `nas-agent` project (the directory containing `nas_agent/` and `pyproject.toml`). The working directory is `<output_dir>`, not the project root, so resolve `<nas_agent_root>` once by probing the installed package:
  ```bash
  python -c "from pathlib import Path; import nas_agent; print(Path(nas_agent.__file__).resolve().parent.parent)"
  ```
  Treat the printed absolute path as the resolved value of `<nas_agent_root>` throughout this skill (e.g. `ruff check --fix --config <nas_agent_root>/nas_agent/internal_ruff.toml`).
- **Path handling**: Use `pathlib.Path` objects for path construction instead of string concatenation or `os.path.join`. Example:
  ```python
  from pathlib import Path

  output_dir = Path(args.output_dir)
  output_dir.mkdir(parents=True, exist_ok=True)
  checkpoint_path = output_dir / "supernet_latest.pth"
  ```

## Workflow

### Step 1: Load Context

1. **Read the upstream summary:** Read `<output_dir>/supernet_summary.md`. Extract the source project path, model type, pre-built block info, task context, and the list of previously generated artifacts.
2. **Read generated artifacts:** Read `<output_dir>/supernet.py` and `<output_dir>/inspect_supernet.py` to understand the supernet architecture, `SearchSpace`, and how the supernet is structured.
3. **Read user training code:** Read the user's training code under `<user_project_root>`, focusing on:
   - Training loop structure and flow
   - Data pipeline (dataset loading, transforms, dataloaders)
   - Optimizer and learning rate scheduler configuration
   - Loss function(s) and metrics
   - Checkpoint saving/loading conventions
   - Any domain-specific training patterns (e.g. iterative solvers, multi-stage training)

### Step 2: Generate Supernet Training Scripts

First, determine whether supernet training is viable for the user's task. Supernet training requires that the training loop supports forwarding different sub-network configurations through shared weights and obtaining a meaningful differentiable loss for each — whether using sandwich sampling (max, min, and random configs per batch) or any other sampling strategy.

Supernet training is **not viable** when:

- Training is driven by environment interaction or non-differentiable reward/metric signals (e.g. reinforcement learning with policy gradient, MCTS-guided training, sequence-level reward optimization with REINFORCE).
- Training involves multi-model co-optimization where each model's loss depends on the other model's current outputs (e.g. GAN generator/discriminator).
- The loss depends on a self-referential copy of the model that would be invalidated by per-batch subnet switching (e.g. momentum encoder in MoCo, stop-gradient branch in BYOL/DINO).

**If supernet training is viable:**

Read `<skill_dir>/references/workflows/train_supernet_script_generation.md`. Follow it to generate project-specific training scripts from the supernet and the user's own training code.

This path generates `train_supernet.py`, `run_train_supernet.sh`, and any small helper files needed by the generated training workflow. After generation:

1. **Run functional smoke tests** defined in the workflow's Validation section. Fix any failures before proceeding.
2. **Invoke the `workflow-verifier` subagent** with:
   - **Workflow**: `<skill_dir>/references/workflows/train_supernet_script_generation.md`
   - **Artifacts** (verifier may modify): `train_supernet.py`, `run_train_supernet.sh`, and any generated helper files
   - **Cross-references** (read-only): The user's original training script from `<user_project_root>` and `<output_dir>/supernet_summary.md` so the verifier can verify artifacts against the original logic.
3. **Handle the verifier response:**
   - If `all-pass` and the response has no **Fixed** section: proceed to Step 3.
   - If `all-pass` and the response includes a **Fixed** section: re-run the functional smoke tests defined in the workflow's Validation section before proceeding to Step 3.
   - If `unresolved`: read each unresolved item, apply the suggested fix in the artifacts, then re-run the functional smoke tests defined in the workflow's Validation section before proceeding to Step 3.

**If supernet training is not viable:**

Skip script generation. Proceed to Step 3.

### Step 3: Complete Summary and Present Next Steps

Read `<skill_dir>/references/evaluation_paradigm.md` before starting this step. Apply its decision logic using the project context gathered in Step 1 and the viability decision from Step 2 to determine the evaluation paradigm.

1. **Update `supernet_summary.md`:** Open the existing `<output_dir>/supernet_summary.md` and add/update the following sections:

   - **Supernet Training Viability**:
     - `viable`: `Yes` or `No`.
     - `reason`: brief project evidence for the viability decision.
     - When `No`, include the following note verbatim:

       > [!WARNING]
       > Supernet training is not viable for this project. `train_supernet.py` and `run_train_supernet.sh` were not generated. The search pipeline will evaluate candidates by training each subnet from scratch instead of relying on a trained supernet checkpoint.
   - **Evaluation Paradigm**: state the chosen paradigm (`validate`, `finetune`, or `train_from_scratch`) and cite the project evidence that led to this choice. When the paradigm is `finetune` due to cross-dataset evaluation, explicitly note that the supernet was pretrained on a source dataset and the search must evaluate subnets on a different target dataset.
   - **Knowledge Distillation**: when supernet training is viable, state whether KD was enabled in the generated `train_supernet.py` (`Yes` / `No`) and briefly explain why. When supernet training is not viable, state `N/A`.
   - **Task And Training Context** (update the existing section if needed):
     - When supernet training is not viable, also include:
       - The user's original training loop structure (e.g. RL rollout collection, GAN G/D alternation, self-supervised momentum update).
       - The original training code file path.
       - The specific reason supernet training does not apply.
     - When the supernet is pretrained on one dataset and the search targets a different dataset, also record the source and target dataset details (names, paths or loading instructions, class counts, preprocessing differences) so the downstream search pipeline can configure the evaluator data loaders correctly.
   - **Generated Artifacts**: append any new files generated in Step 2 (e.g. `train_supernet.py`, `run_train_supernet.sh`) to the existing list.

2. **Warn if supernet training is not viable:** When `Supernet Training Viability` is `No`, explicitly warn the user in the conversation. Briefly state the reason and refer to the Supernet Training Viability section in `supernet_summary.md` for details.

3. **Present next steps to the user:** Briefly recap the key decisions (paradigm, viability, KD) and inform the user that the next step is to generate the NAS search pipeline scripts in a new session using the `nas-search-pipeline` skill.

## Validation

- For generated remote-execution scripts such as `train_supernet.py` and `run_train_supernet.sh`, follow the specific validation contract in the referenced workflow, including static checks and functional smoke tests. These scripts must not be executed as a full training run locally.
- If validation fails, fix the artifact and rerun the same validation before proceeding.

## Guidelines

- Preserve all generated artifacts unless the user explicitly asks for cleanup.
- Keep standalone model files free of `ModuleNotFoundError` for local project code.
- Prefer conservative recommendations when task, data, or deployment context is uncertain.
- Keep generated Python variable names, function names, classes, string literals, comments, and docstrings in English.
