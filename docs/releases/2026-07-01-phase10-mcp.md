## 概述

phase 10 完成 Orca 的第三个壳 —— **MCP 壳**（外部 MCP 服务）。任意 MCP 兼容客户端
（Claude Code / opencode / Cursor / Windsurf / Cline / Zed）现在可以通过 stdio JSON-RPC
把 Orca workflow 当工具调，长跑 gate 不超时，与 Web 壳同进程共存零漂移。

五件交付：

1. **HandleId 四件套工具**（start_workflow / get_task_status / resolve_gate / cancel_task）—— 每 tool 秒级返回，
   规避 CC 60s 硬超时。
2. **tape-only query path** —— `pending_gates_from_tape` / `run_summary` 从 tape 派生，
   不读 `HumanGateHandler` 内存（防 AgentHarness 多真相源重蹈）。
3. **单进程多壳共存** —— `orca mcp [--with-web]`，MCP（stdio）+ Web（HTTP）共享同一 `RunManager` 实例。
4. **跨客户端兼容** —— stdio 每消息 flush 兜底（规避 opencode #21516），不依赖 elicitation / progress notification。
5. **5 个 E2E 闭环测试** —— 真 Orca workflow（demo_linear / 慢 script / demo_task）+ 真引擎 + 真 stdio round-trip。

## 架构决策（防 drift）

### 1. 单进程单 RunManager（SPEC §0.1 第一条 / §1.4）

```
┌─────────────── 一个 Orca 进程（orca mcp [--with-web]）────────────────┐
│   RunManager（唯一实例，进程级单例，gc.get_objects 启动 assert）       │
│           ↑ resolve（入）              ↓ events（出）                    │
│    ┌──────┴──────────┐            ┌──────────┴──────────┐              │
│    │ HumanGateHandler │            │      EventBus        │             │
│    └──┬────────┬──────┘            └─────┬──────────┬─────┘              │
│       │        │                         │          │                    │
│   ┌───┴──┐ ┌───┴────┐            ┌──────┴──┐  ┌─────┴──────┐            │
│   │ MCP  │ │ Web HTTP│            │ MCP tool│  │ Web WS pump│            │
│   │source│ │source=  │            │ 返回值  │  │ push events │           │
│   │="mcp"│ │"web"    │            │         │  │             │           │
│   └───┬──┘ └────┬───┘            └─────────┘  └─────────────┘            │
└───────┼─────────┼──────────────────────────────────────────────────────┘
        │         │
   stdio       HTTP/WS
```

跨进程共享 in-memory state 是反模式（live run owner 唯一，第二进程只能读 tape 做
read-only replay，连 gate 都 resolve 不了）。启动 assert + 单测构造二实例检测保护此铁律。

### 2. HandleId pattern（SPEC §0.1 第三条 / 草稿 §5.2）

每个 MCP tool 秒级返回，server 端不阻塞等 gate / 等事件（CC 60s 硬超时绝对禁忌）。

- `start_workflow` 复用 `manager.start_run`（已有，非阻塞后台 task）
- `get_task_status` 读 tape（`replay_state` + `pending_gates_from_tape`）
- `resolve_gate` 复用 `handler.resolve`（同步非阻塞，first-wins）
- `cancel_task` 复用 `manager.cancel_run`（写 tape + cancel asyncio task）

`test_start_workflow_does_not_block_on_slow_orchestrator` 真测：用真 RunManager + 慢
orchestrator（asyncio.sleep 5s），断言 tool 在 1s 内返回。

### 3. tape-only query path（SPEC §0.1 第二条 / §3 ⭐ 核心）

**任何对外查询的唯一数据源是 tape**。`HumanGateHandler._pending` / `_gates_meta` 是 runtime
await 状态，禁止作为查询路径。新增三个派生函数：

| 函数 | 位置 | 派生自 |
|---|---|---|
| `replay_state(tape)` | `orca/events/replay.py`（已有）| tape events fold |
| `pending_gates_from_tape(tape)` | `orca/gates/pending.py`（**新**）| tape events（requested - resolved）|
| `RunManager.run_summary(run_id)` | `orca/iface/web/run_manager.py`（**新方法**）| merge meta + state + pending_gates |

`test_does_not_touch_handler_internal_state` 反例断言：mock handler 不存在仍能派生正确。

### 4. source="mcp" 复用 handler.resolve（SPEC §0.1 第五条）

MCP resolve 走 `HumanGateHandler.resolve` 同一入口（phase-6 + phase-9a 已稳定），
`source="mcp"` 写死。零新 resolve 路径，竞速 first-wins + `_broadcaster` 广播与 Web 同款。
**E2E-2 端到端断言**：`handler.request` 收到的 `(answer, source)` 中 source 必须是 "mcp"。

### 5. stdin EOF 双行为（SPEC §1.3）

- 无 `--with-web`：stdin EOF → drain + cancel 后台 task → 进程退出（随 CC session 生灭）
- 有 `--with-web`：stdin EOF → 转 daemon 模式 → 退出条件：SIGINT/SIGTERM 或无活跃 run 持续 `idle_timeout` 分钟

## 偏离 SPEC/plan 的地方（透明记录）

### 1. cancel_run emit 顺序调整（D1）
- **SPEC §5.3 伪代码**：`task.cancel()` 先，`bus.emit("workflow_cancelled")` 后
- **实际实现**：emit 先，task.cancel 后
- **理由**：`task.cancel()` 调度 task 的 finally teardown（`_teardown_handle` → `bus.close()`），若 emit 在 cancel 之后可能 race，emit 时 Tape 已 close → RuntimeError。emit-first 保证 cancel 事件落 tape 后才允许 teardown。**Hard invariant「emit before status=cancelled」仍满足**，只是 task.cancel 的相对顺序后移。docstring 已注明。

### 2. mcp 依赖锁 `<1.28` + `cryptography<49`（D2）
- **SPEC §6.5 要求**：`mcp>=1.0`
- **实际**：`mcp>=1.0,<1.28` + `cryptography<49`
- **理由**：mcp 1.28.1+ 经 `pyjwt[crypto]` 依赖 cryptography 49，cryptography 49 在 darwin x86_64 无预编译 wheel（需 rust/maturin 本地构建，CI 不友好）。锁 `<1.28` + `<49` 走预编译 wheel。注释已说明，待 cryptography 49 提供更广平台 wheel 可放宽。

### 3. E2E-2/3 用慢 script（sleep 10）而非 demo_linear（D5）
- **plan §D5.3 暗示**：用 `demo_linear.yaml`
- **实际**：用临时慢 script workflow（单节点 sleep 10）
- **理由**：demo_linear 三节点 echo 秒级完成，orchestrator 完成后 `_teardown_handle` 关 tape，测试侧 fire gate 写 tape 会 `RuntimeError: Tape 已 close`。慢 script 让 orchestrator 在 sleep 期间保持 tape 活，**不 mock 引擎**（仍真 Orchestrator + 真 tape + 真 EventBus）。conftest `slow_workflow` docstring 详述。

### 4. start_workflow 不阻塞测试用真 RunManager 而非 mock manager（D3）
- **plan §D3.5 暗示**：mock `manager.start_run` 内 sleep 5s
- **实际**：真 RunManager + monkey-patch `_run_with_sem` 慢编排
- **理由**：原 plan 测法在概念上错——start_workflow 直接 await start_run，若 start_run 阻塞 tool 跟着阻塞，mock 无法证明 non-blocking。改用真 RunManager + 慢编排证明 tool 不等编排完成（这才是 HandleId 真意图）。

### 5. lifecycle 第三种 stdin EOF 改 mock 单测（D4）
- **plan §D4.4 要求**：subprocess 真测 `--with-web` + 活跃 run + stdin EOF 不退出
- **实际**：逻辑由 `test_wait_for_idle_or_signal_returns_when_idle_immediately` 单测（patch asyncio.sleep）覆盖
- **理由**：60s+ 真等 daemon tick CI 太慢。逻辑分支由 mock 单测完全覆盖。release note 注明。

### 6. orca mcp 加 `--runs-dir` flag（D5）
- **SPEC/plan 未提**
- **实际**：`commands.py` mcp subcommand 加 `--runs-dir` 选项
- **理由**：subprocess E2E fixture 必须隔离 tape 落盘（不污染项目 `runs/`）。`run_mcp_server` 已支持 `runs_dir` 参数（D2），CLI 只是把口子打开。业务用户无需配置（默认 `./runs`）。
