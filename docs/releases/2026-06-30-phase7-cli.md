## 做了什么

实现 Orca 的第一个端到端入口：``orca run/validate/list`` CLI + Textual TUI（DAG 进度
面板 + 流式日志 + gate ModalScreen）。用户在终端 ``orca run nas.yaml`` 就是一个可用的
编排工具。

### 关键架构决策：gate 触发的双线程拓扑

orchestrator 对 gate 透明（phase 6 §2.3）。两个触发来源：

1. **tool_permission**：claude 的 PreToolUse hook POST 到 `/gate`。CLI 必须跑一个后台
   HTTP server（FastAPI + uvicorn，复用 phase 6 的 `register_gate_routes`），共享同一份
   `HumanGateHandler` + `SessionContextRegistry` + `EventBus`。
2. **agent_ask**：`ask_user(handler, ...)` 在 node 执行内调（不需 HTTP）。

**决策**：uvicorn 跑在**独立 daemon 线程 + 独立 asyncio loop**（`_GateHttpBridge`）。
共享对象（`HumanGateHandler` / `SessionContextRegistry` / `EventBus.tape`）均用
`threading.Lock` 保护跨线程安全，故 TUI 的 loop 与 HTTP server 的 loop 可安全并发访问。
- 避免「两个 asyncio loop 抢一个线程」的崩溃。
- 避免 uvicorn 嵌入 Textual 的兼容性坑（`Server.serve` 必须在其 loop 内 await）。
- loop 隔离 + 共享对象锁 = 干净的双线程拓扑。

socket **预 bind**（在创建 uvicorn server 前同步 bind+listen）：① 端口占用 deterministic
立即失败（不依赖 uvicorn 内部 startup 异步时序）；② 消除「假就绪」窗口。

gate ModalScreen 双重身份（SPEC §4.5 决策 5）：
- `human_decision_requested` → `_push_gate_modal` 是独立 `@work` worker，`push_screen_wait`
  阻塞该 worker 但 UI 事件循环继续刷新 DAG/日志（Textual 决定性优势，专项回归测试守护）。
- `human_decision_resolved`（别壳先答）→ `notify_resolved_externally` 让 modal dismiss，
  返回 `__orca_broadcast__:<source>:<answer>` 哨兵；OrcaApp 识别哨兵后**不重复 resolve**
  （赢家已 resolve）。赢家路径调 `handler.resolve(gate.id, answer, "cli")`。

### surgical bug 修复：hook_script.py sys.path 阴影

`python orca/gates/hook_script.py` 直接运行时，Python 把脚本目录 `orca/gates/` 加到
`sys.path[0]`，导致 `import json` → `import re` → `from types import GenericAlias` 把
`orca/gates/types.py` 误当 stdlib `types`（circular import）。+10 行 surgical 修复：在
所有其他 import 前，若 `sys.path[0]` 在 `orca/gates/` 下则摘除。这是 phase 6 hook 桥
跑不起来的根因（clean master 上 9 个 test_hook_bridge 测试 fail），fold 进 phase 7 修复
以达成「全套绿」的验收目标。**不改任何 hook 语义**（exit 0/2 / HTTP / 超时逻辑零改动）。

## 偏离计划

- 计划 C3 的 `on_mount` 起编排 worker + HTTP 桥 → 改为分离的 `kickoff()` 方法
  （SRP：on_mount = 组件就绪，kickoff = 业务开始）。理由：headless pilot 测试需控制编排
  /HTTP 桥启动，不分离则每个单测都会真跑 Orchestrator（spawn claude/script）+ uvicorn。
- 计划 C3 没写 `_GateHttpBridge` 的并发测试 → review 后补了 2 个（clean start/stop +
  occupied-port fail loud）。
- 计划 C1 没写 `orca run` completed→0 的 headless 测试 → review 后补了 3 个退出码测试
  （completed→0 / failed→1 / 用户 q 退出→1，mock Orchestrator 不依赖真 claude）。

  5. Textual（非 Rich Live）。
- **`orca --help`**：显示 run/validate/list 子命令。
