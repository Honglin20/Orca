## 1. 做了什么

实现 run/ 编排层 —— 让校验过的 Workflow 真正跑起来。单指针推进 + parallel 组并行
+ foreach 分批 + 路由 first-match-wins + 循环终止，事件全程 `bus.emit` 落 Tape。

## 2. 关键设计决策

### 2.1 ctx.outputs 存储形状：`{"output": raw}`

orchestrator 把 executor 返回的 raw output 包装成 `{"output": raw}` 存进 `ctx.outputs[node]`。
这是 render.py `_namespace` 的约定（模板用 `{{ node.output.field }}`，从 `outputs[node]["output"]` 取）。
parallel/foreach 返回 raw aggregated，由 orchestrator 统一包装（不重复包）。
**reducer 的 RunState.context 存的是 raw output（非包装）** —— last-writer-wins 直接存 `data["output"]`。

### 2.2 RunContext 扩展方式：加 `locals` 字段（非 inputs 注入）

foreach body 模板用 `{{ item }}`/`{{ _index }}` 裸引用（非 `{{ inputs.item }}`），所有 example
都这么写。给 RunContext 加 `locals: dict` + `with_locals(locals_)` 方法（返回新 frozen 实例），
render 的 `_namespace` 摊 locals 到顶层。这是 phase 4 render.py 注释「foreach body 注入（phase 5）」
预示的扩展 —— 加的是数据字段，编排逻辑仍在 run/，exec 仍不 import run，**不算职责越界**。

### 2.3 emit `route_taken` 事件

orchestrator 每步路由后 emit `route_taken(from, to)`，让 reducer 的 `current_node` 正确跟踪
（replay.py:109 据 `data.to` 更新 current_node）。否则 tape 事件流不完整、replay 的 current_node 错。

### 2.4 bus.emit 真实签名：`(type, data, node, session_id)`

executor 产出的是 Event 对象（seq=0 占位），但 `bus.emit` 吃拆解后的四参（不是预构造 Event）。
seq 由 `tape.append` 在 emit 内部重分配（占位 seq=0 被覆盖）。故 `executor_adapter` 必须把每个
executor Event 拆成 `(event.type, event.data, event.node, event.session_id)` 再调 `bus.emit`。

### 2.5 failure_mode 聚合共享 `aggregate.decide_failure`（DRY）

parallel 组和 foreach 的 failure_mode 决策逻辑相同（fail_fast / continue_on_error /
all_or_nothing 三态），抽到 `aggregate.py` 共享。结果容器形状不同（parallel: `{outputs: {branch: raw}}`；
foreach: `{outputs: [raw...]}`），但失败判定相同。

**fail_fast 语义说明**：由于上层用 `asyncio.gather(return_exceptions=True)` 等全部完成，
fail_fast 在并行场景退化为「全部完成后抛首个失败」（与 all_or_nothing 行为等价）。
真正的「不等其余」需用 `asyncio.wait(FIRST_EXCEPTION)` + 取消，超出本阶段范围。

### 2.6 foreach source 求值：手写点路径（避开 Jinja2 dict 方法冲突）

Jinja2 的点访问对 dict 优先取属性（`dict.items` 是 builtin method，与名为 `items` 的 key 冲突）。
故 `_eval_source_array` 手写点路径遍历，每跳优先 `dict[key]`（`__getitem__`），避开冲突。
另支持 set 节点产出的字符串化数组（`"[1,2,3]"` → JSON 解析回 list）。

### 4.1 集合 bug 修复（follow-up）

`tests/run/test_{orchestrator,parallel,foreach}.py` 用 `from tests.run.conftest import ...`
引用共享 helper（FakeExecutor / make_bus / run_async），但 `tests/` 不是包（无
`tests/__init__.py`）→ `ModuleNotFoundError: No module named 'tests.run'`，三个测试文件
collection 失败。**修复**：新增空 `tests/__init__.py`（与既有 `tests/{run,exec,compile,
events,profiles}/__init__.py` 一致；`tests/run/conftest.py` 注释同步澄清）。

修复选择理由：`tests/exec/conftest.py` 既有约定是「tests 非包 → helper 就地复制」，
但 `tests/run/` 已有 `__init__.py` 且 conftest 注释自陈「tests/run 是包，可 import」——
即 5-R 的**意图**就是用包导入。补 `tests/__init__.py` 是修复根因（让 `tests.run` 真正可
绝对导入），而非引入更脏的 helper 复制。零回归（`tests/schema` 无 `__init__.py` 仍由
pytest rootdir 插入正常 collection）。

## 5. 偏离 SPEC/计划的判断

1. **计划伪代码 `bus.emit(event, session_id=...)` 错**：实际签名是 `emit(type, data, node, session_id)`，
   adapter 按真实签名拆解。已在执行前核实 bus.py:117。

   改用 `is defined` 守卫自引用上轮 `counter.output.n`，实现真递增。
3. **demo_foreach 的 source**：计划用 `items` 作 key，但与 dict.items() 方法冲突。foreach 求值器
   手写点路径避开（仍支持 `items` key 名）。

   对前端定位失败 node 必要，扩展 exec.error.py 加可选 node 字段（向后兼容）。
