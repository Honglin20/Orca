## 改动点

### server.py 重写：6 旧工具 → 9 v4 工具

旧 6-tool 设计（含 `resolve_gate` / yaml_path-based `start_workflow`）按 v4 SPEC 全量重写：

- **Discovery 4**：`list_workflows` / `describe_workflow` / `list_agents` / `get_agent_prompt`
- **Lifecycle 3**：`start_workflow`（name-based + `setup_outputs` 参数）/ `get_task_status` / `cancel_task`
- **History 2**：`get_task_history` / `get_agent`

删 `resolve_gate`（铁律 7：execute phase 永不中断，无 needs_decision 状态）。

### Result 信封（ADR §4.1）

所有 9 工具返回 `Result` 信封，序列化为 `{ok, data?, error?, _hint?}`：
- `error.kind` 是 `ErrorKind` 值（`business_config` / `transport_network` / ...）
- **无 `layer` 字段**（ADR §4.1 决策 1.3：kind 前缀派生 layer）
- setup_required / setup_outputs_mismatch / setup_outputs_invalid → `kind=business_config` + 描述 message + 引导 _hint
- MCP 边界不重新分类 run 层 kind（ADR §4.1 决策 1.4）

### setup/execute 分相

- `Workflow.setup: list[AgentNode] = []`（可选，空 = 无 setup phase）
- compile validator 加 `_check_execute_phase_no_gate_tools`：execute phase（`wf.nodes`）AgentNode 显式配 `ask_user` / `gate` tools → fail loud
- setup phase agent 不受此限（允许 ask_user / gate，TUI/Web 实际跑）
- Jinja2 valid roots 加 `setup`（`{{ setup.<agent>.output.<field> }}`）

### 三重杠杆防跳过 setup（§2.8）

- **杠杆 A**：`list_workflows` 返 `has_setup` 标记
- **杠杆 B**：`start_workflow` 检测 `setup_required` → `kind=business_config` + 引导 _hint
- **杠杆 C**：tool description 显式写 "must describe first"

## 偏离 SPEC 的地方

1. **start_workflow 仍走 RunManager.start_run(yaml_path)**：v4 SPEC §2.7 设想 RunManager.start 接受 `setup_outputs` 并注入 context。但本任务不动 RunManager（并行工作约束），setup_outputs 只在 MCP 层校验，不注入 workflow runtime。compile validator 允许 `setup.*` jinja root，但 render 层未注入 `setup` namespace。**技术债**：待 RunManager 扩展后回填 setup_outputs 注入。
2. **删 test_e2e_gate.py / test_e2e_cross_shell.py**：这两个 E2E 测 `resolve_gate`（v4 删），直接删除。未来 agent-interrupt feature 落地后可重建 MCP-based interrupt E2E。

## 遗留技术债

1. **CLI 挂载**：`orca mcp` subcommand 注册在 `executor_cmds.py`（dirty，并行工作持有），本任务不动。CLI 仍用旧 6-tool 注册（v4 server.py 已就位，CLI 挂载待并行工作落地）。
2. **RunManager 扩展**：`run_summary` / `list_runs` / `cancel_run` 已存在（phase-9a 落地），不需扩展。但 `start_run` 不接受 `setup_outputs`（setup phase injection 未实现）。
3. **phase-12 capability 校验集成**：compile validator 的 `_check_execute_phase_no_gate_tools` 是简单 tools 白名单检查，与 phase-12 CapabilitySet 正交。未来 CapabilitySet 落地后可增强为基于 capability 的检查。
4. **`workflow_cancelled` 事件类型**：已在 phase-9a 落地（RunManager.cancel_run emit），本任务复用。
5. **setup_outputs runtime injection**：`setup` jinja root 在 compile 层已合法，但 render 层未注入 `setup` namespace（render.py 不变，属 RunManager 扩展范围）。
